# 计费分账引擎

> 核心中的核心，所有变现收入的归因和分配都经过这个引擎

---

### BIL-001 计费事件Schema定义
- **优先级**: P0
- **预估工时**: 2天
- **依赖**: 无
- **描述**: 定义所有可计费事件的数据结构，包括播放事件、广告事件、付费事件三大类。每类事件包含公共字段（event_id, user_id, device_id, timestamp, session_id）和业务字段。
- **技术要点**:
  - 播放事件：drama_id, episode_id, play_duration, is_valid_play, completion_rate, quality, network_type
  - 广告事件：ad_id, ad_type(pre/mid/post), event_type(impression/click/complete), ecpm, revenue
  - 付费事件：order_id, product_type(episode/drama/membership), amount, channel_fee, tax, status
  - 使用 Protobuf 或 JSON Schema 定义，确保序列化/反序列化一致性
- **涉及文件**: `backend/billing/schemas/`, `backend/billing/proto/`
- **验收标准**:
  - [ ] 三类事件 Schema 定义完整
  - [ ] 字段说明文档齐全
  - [ ] 序列化/反序列化测试通过

### BIL-002 事件采集管道
- **优先级**: P0
- **预估工时**: 3天
- **依赖**: BIL-001, INF-003
- **描述**: 搭建基于 Kafka 的事件采集管道。App 端和后端产生的计费事件统一写入 Kafka Topic，Consumer 负责清洗、去重、持久化。
- **技术要点**:
  - Kafka Topic 设计：billing_play_events, billing_ad_events, billing_pay_events
  - Producer：后端服务通过 SDK 写入，保证 at-least-once
  - Consumer：消费事件，写入计费数据库（ClickHouse/MySQL）
  - 去重机制：event_id 幂等去重（Redis Set + TTL）
  - 延迟容忍：允许 T+1 补报
- **涉及文件**: `backend/billing/pipeline/`, `backend/billing/consumer/`
- **验收标准**:
  - [ ] 事件从产生到入库延迟 < 5秒（实时流）
  - [ ] 重复事件自动去重
  - [ ] 补报事件正确处理
  - [ ] 消费积压监控告警

### BIL-003 事件归因逻辑
- **优先级**: P0
- **预估工时**: 3天
- **依赖**: BIL-001, BIL-002
- **描述**: 将每一条计费事件归因到具体的内容（剧/集）和合作方（partner_id），建立"事件→内容→合作方"的完整归因链路。
- **技术要点**:
  - 维护 episode_id → drama_id → partner_id 的映射表（缓存在 Redis）
  - 处理内容转移场景（合作方变更、版权到期）
  - 归因时间窗口：以事件发生时间为准，查当时的归属关系
  - 异常处理：找不到归属的事件进入人工审核队列
- **涉及文件**: `backend/billing/attribution/`
- **验收标准**:
  - [ ] 100% 事件可归因到合作方
  - [ ] 版权变更时归因正确切换
  - [ ] 异常事件进入审核队列

### BIL-004 分账规则引擎核心
- **优先级**: P0
- **预估工时**: 5天
- **依赖**: BIL-003
- **描述**: 构建可配置的分账规则引擎，支持按合作方、内容、时间段配置不同的分账规则。规则可组合、可版本化、可回放。
- **技术要点**:
  - 规则模型：Rule(id, partner_id, drama_id, rule_type, params_json, version, effective_from, effective_to)
  - 规则匹配：优先级 drama级别 > partner级别 > 全局默认
  - 计算引擎：输入（事件聚合数据）→ 应用规则 → 输出（各方分成金额）
  - 支持多规则叠加（如 IAA + IAP 混合分账）
  - 沙盒模式：新规则可先模拟计算不实际生效
- **涉及文件**: `backend/billing/engine/`, `backend/billing/rules/`
- **验收标准**:
  - [ ] 规则 CRUD 接口完整
  - [ ] 规则匹配优先级正确
  - [ ] 沙盒模式可模拟计算
  - [ ] 规则变更有审计日志

### BIL-005 IAA分账规则（广告收益分成）
- **优先级**: P1
- **预估工时**: 3天
- **依赖**: BIL-004
- **描述**: 实现基于广告收益的分账规则。按有效播放量阶梯定价 + 广告实际收益分成两种模式。
- **技术要点**:
  - 阶梯模式：0-10万播放 ¥X/千次，10-50万 ¥Y/千次，50万+ ¥Z/千次
  - 收益分成模式：广告实际收入 × 分成比例（如70%给合作方）
  - 有效播放定义：播放>=30秒 且 非作弊流量
  - 完播加权：完播率>80%的播放权重×1.2
  - 支持按广告类型（前贴/中插）设置不同分成
- **涉及文件**: `backend/billing/rules/iaa_rule.py`
- **验收标准**:
  - [ ] 阶梯计算结果正确
  - [ ] 广告收益分成计算正确
  - [ ] 完播加权生效
  - [ ] 无效播放自动剔除

### BIL-006 IAP分账规则（付费分成）
- **优先级**: P1
- **预估工时**: 3天
- **依赖**: BIL-004
- **描述**: 实现基于用户付费的分账规则。按实际收入扣除渠道费、税费、退款后，按比例分成。
- **技术要点**:
  - 计算公式：分成金额 = (实收 - 渠道费 - 税费 - 退款) × 分成比例
  - 渠道费：Apple 30%/15%（小企业计划），自动识别
  - 税费：增值税6%（可配置）
  - 退款扣回：退款订单在结算时自动扣除
  - 支持单集/全集/会员不同分成比例
- **涉及文件**: `backend/billing/rules/iap_rule.py`
- **验收标准**:
  - [ ] 渠道费自动计算
  - [ ] 税费正确扣除
  - [ ] 退款自动扣回
  - [ ] 不同商品类型分成比例独立配置

### BIL-007 会员分账规则（订阅收入分摊）
- **优先级**: P1
- **预估工时**: 4天
- **依赖**: BIL-004
- **描述**: 会员订阅收入按"收入池"模式分配。每月会员总收入扣除平台成本后，按各合作方内容的有效观看时长/完播集数加权分摊。
- **技术要点**:
  - 收入池 = 当月会员总收入 - 渠道费 - 税费 - 平台运营成本（可配置比例）
  - 权重计算：合作方权重 = Σ(该方内容有效观看时长) / 全平台总有效观看时长
  - 分摊金额 = 收入池 × 权重
  - 支持完播集数权重模式（替代时长模式）
  - 最低保障：若分摊结果低于保底则按保底计算
- **涉及文件**: `backend/billing/rules/membership_rule.py`
- **验收标准**:
  - [ ] 收入池计算正确
  - [ ] 权重分摊公平合理
  - [ ] 保底机制生效
  - [ ] 支持时长/完播两种权重模式切换

### BIL-008 保底+超额分成模式
- **优先级**: P2
- **预估工时**: 2天
- **依赖**: BIL-004
- **描述**: 对优质合作方提供保底分成 + 超额部分更高比例分成的激励模式。
- **技术要点**:
  - 保底金额：月保底 ¥N（不论实际收益是否达到）
  - 超额分成：超过保底的部分，合作方拿更高比例（如80%）
  - 合同级别配置：每个合作方可设不同保底条款
  - 亏损预警：保底持续高于实际收益时告警
- **涉及文件**: `backend/billing/rules/guaranteed_rule.py`
- **验收标准**:
  - [ ] 未达保底时按保底金额结算
  - [ ] 超额部分按配置比例计算
  - [ ] 亏损预警正常触发

### BIL-015 退款/作弊扣回机制
- **优先级**: P1
- **预估工时**: 3天
- **依赖**: BIL-013
- **描述**: 在结算周期内发生的退款和作弊行为，自动从合作方结算中扣回相应金额。
- **技术要点**:
  - 退款扣回：支付退款后自动标记对应计费事件为无效，结算时扣除
  - 作弊扣回：反作弊系统标记的无效播放/点击，结算时扣除
  - 跨周期扣回：上一周期的退款在本周期结算中扣除
  - 争议处理：合作方可对扣回金额提出申诉
  - 扣回上限：单周期扣回不超过结算金额的30%（剩余转下期）
- **涉及文件**: `backend/billing/deduction/`
- **验收标准**:
  - [ ] 退款自动触发扣回
  - [ ] 作弊标记正确扣回
  - [ ] 跨周期扣回正确
  - [ ] 扣回上限保护生效
